## Mejores prácticas para mensajes de commit:

Un mensaje de commit debe indicar: propósito del cambio, motivo del cambio, proceso que afecta. 
Si puedes, escribe una sola oración para indicar todo, pero si no puedes, usa una oración de resumen y luego un parrafo o listado

---
### Prefijos sobre el propósito del commit
Usa un prefijo al inicio del mensaje que indique tipo de commit:
- improvement: se mejora un proceso ya existente
- feat: Una nueva caracteristica. ("feat" es la abreviacion de "feature" = Característica)
- fix: Se soluciono un bug.
- docs: Se realizaron cambios en la documentacion.
- style: Se aplico formato, comas y puntos faltantes, etc; Sin cambios en el codigo.
- refactor: Refactorizacion del codigo en produccion. (refactorizacion = mejorar codigo sin cambiar/agregar funcionalidad)
- test: Se añadieron pruebas, refactorizacion de pruebas; Sin cambios en el codigo.
- chore: Actualizacion de tareas de build, configuracion del admin. de paquetes; Sin cambios en el codigo.
- wip: work in progress (trabajo en construccion)
- delete: se borran archivos (ejemplo, se borran archivos log. Tambien conocido como "cleanup")
- security: cambios con propósito de mejorar la seguridad del software o correjir una vulnerabilidad, ya sea por simplemente aumentar la version de algun package o tecnología base
